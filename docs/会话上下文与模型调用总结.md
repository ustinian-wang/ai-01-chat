# ai-01-chat：会话、上下文与模型调用总结

本文档归纳本项目中与「消息存储」「上下文预算」「如何调用第三方模型」「Chat API 角色」相关的实现与约定，便于查阅与交接。启动方式、环境变量清单仍以仓库根目录 `README.md` 为准。

---

## 1. 消息持久化策略（`message_store`）

| 维度 | 说明 |
|------|------|
| 隔离键 | 按 `thread_id` 区分会话；磁盘路径为 `backend/data/sessions/<安全化 thread_id>.json`（非法字符替换为 `_` 等，防路径穿越）。 |
| 内存 | 进程内 `_store: dict[thread_id, list[消息]]`，首次访问某会话时从磁盘装入。 |
| 磁盘格式 | JSON **信封**：`thread_id`、`title`、`updated_at`、`messages`；兼容旧版「根节点为纯数组」的文件。 |
| 写入 | `append_exchange` 在内存末尾追加一轮 `user` + `assistant` 后 **整文件重写**；非单条 patch。 |
| 列表 | `list_sessions` 扫描目录下 `*.json`，按文件 **mtime 倒序**；标题优先文件内 `title`，否则取首条用户内容摘要。 |
| 空会话 | `ensure_session` 可写入空 `messages`，便于左侧栏立即出现「新对话」。 |
| 删除 | `clear_thread`：内存 `pop` + 删除对应 JSON 文件。 |
| 并发 | 公开 async API 使用 **`asyncio.Lock`** 串行化读写。 |

相关代码：`backend/app/services/message_store.py`。

## 2. 上下文过长时的处理（请求级截断）

- **磁盘**：始终保留全量历史（除非用户删除会话）。
- **调用模型前**：从 `prior` 列表 **头部（最旧）** 逐条删除消息，直到「粗略 token 估算」≤ `CONTEXT_TOKEN_LIMIT - CONTEXT_TOKEN_RESERVE`。
- **估算口径**：与前端 `frontend/src/contextUsage.js` 中 `roughTokenEstimate` 对齐（中日文约 1.5 字/token 等）；估算时将 `system`、历史各条有内容的 `content`、当前用户句用 `\n\n` 拼成一串再估算——**仅用于预算判断，不是发给上游的最终形态**。
- **环境变量**（`backend/.env`）：`CONTEXT_TOKEN_LIMIT`（默认 8192）、`CONTEXT_TOKEN_RESERVE`（默认 1536）。建议与前端 `VITE_CONTEXT_TOKEN_LIMIT` 对齐思路。

相关代码：`message_store.trim_history_for_context` / `estimate_prompt_tokens`；`backend/app/services/openai_stream.py` 在 `build_openai_messages` 之前调用截断。

## 3. 拿到历史后如何「交给模型」（不是拼成一大段字符串）

1. `get_thread_messages(thread_id)` 得到已持久化的消息列表（**不含本轮**用户句；本轮在 HTTP 请求体 `message` 里）。
2. （可选）`trim_history_for_context` 得到截断后的 `prior`。
3. `build_openai_messages(prior, user_display, SYSTEM_PROMPT)` 组装为 OpenAI Chat Completions 所需的 **`messages` 数组**：
   - 首条：`role: "system"`，`content` 为固定 `SYSTEM_PROMPT`；
   - 中间：按顺序追加历史中的 `role` / `content`；
   - 末条：当前轮的 `role: "user"`，`content` 为 `user_display`（若有 `image_url`，仅把说明文字拼进 user 文本，**未做多模态图片入模**）。
4. `AsyncOpenAI().chat.completions.create(model=..., messages=..., stream=True, user=thread_id)` 发往上游。
5. 流式结束后 `append_exchange` 将本轮用户句与助手全文写回存储。

相关代码：`backend/app/services/openai_stream.py`、`message_store.build_openai_messages`。

## 4. 与第三方模型通信（OpenAI 兼容）

- 本服务 **不手写** 上游 HTTP，而是通过 **`openai` Python SDK** 的 **`AsyncOpenAI`** 调用 **`/v1/chat/completions`** 兼容形态。
- **鉴权**：`OPENAI_API_KEY` → `Authorization: Bearer ...`。
- **路由**：若配置 `OPENAI_BASE_URL`（如讯飞星火 HTTP 兼容地址），SDK 向其发送请求；为空则走 OpenAI 默认云端。
- **请求体要点**：`model`、`messages`（见上一节）、`stream: true`、可选 **`user`**（本项目传 `thread_id`）。
- **前端与本服务**：浏览器 `POST /api/v1/chat/stream`，响应为 **自研 SSE**（`data: {"delta":...}` / `done` / `error`）；与上游厂商 SSE 格式由 SDK 消化，不必一致。

## 5. `messages` 里的 `role` 一般有哪些

| role | 含义 |
|------|------|
| `system` | 系统指令 / 人设 / 规则。 |
| `user` | 用户侧内容。 |
| `assistant` | 模型历史回复，用于多轮续写。 |
| `tool` | 工具调用场景下，承载工具执行结果（与 `tool_calls` 等配合）。 |
| `function` | 旧式函数调用角色，部分文档仍可见；新集成以厂商当前文档为准。 |

本项目存储与规范化阶段主要处理 **`system` / `user` / `assistant`**（见 `_normalize_messages`）。
